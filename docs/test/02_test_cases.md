# テストケース一覧

> 環境記号: **E0**=アプリ単体 / **E1**=ローカルDocker / **E2**=Staging / **E3**=Prod

## 0. 事前確認

### 0-A Android: 接続先確認

- **手順**: Logcat または設定画面でベースURLと環境（dev/prod）を確認
- **合格**: 期待する環境・BaseURL と一致している
- **NGの兆候**: データが一切出ない・通知が来ない・保存が効かない、が全部同時に起きる

### 0-B Backend: configs/global と sources

- **手順**: Firestore コンソールで以下を確認
  - `configs/global`: `candidate_target_per_day=200`、`candidate_hard_limit_per_day=500`、`publish_count_per_day=20`、`fallback_max_days=7`
  - `sources`: 30〜60件 `enabled=true`、`priority` が適度にバラけている
- **合格**: 上記フィールドが揃っている

## 1. Android アプリ単体

### A. 起動・認証

**A-01 起動スモーク（E0）**
- 手順: アプリ起動 → ホーム表示まで待つ
- 合格: クラッシュしない

**A-02 匿名ログイン成功（E0/E1/E2）**
- 手順: 起動直後、認証待機 → 完了まで待つ
- 合格: Today の取得に進む（失敗しても再試行導線がある）

**A-03 認証失敗→再試行で復帰（E0/E1/E2）**
- 手順: 機内モード ON で起動 → 失敗を確認 → 機内モード OFF → 再試行ボタンをタップ
- 合格: アプリ再起動なしで復帰する
- NGの兆候: ローディングが永久に続く、画面が真っ白、クラッシュ

---

### B. Today一覧（SC-01）

**B-01 20本表示（E1/E2）**
- 前提: `days/{day_key}` に20本入っている（seed or batch 済み）
- 合格: 20本表示、スクロール快適、サムネ遅延ロードOK

**B-02 バックエンド無し時のUI（E0）**
- 前提: バックエンド無し、または BASE_URL 未到達
- 合格: 画面が壊れない / "取得できません・再試行" 表示 / Pull-to-refresh で再試行できる
- NG: 生のエラーログが画面に垂れ流しになっている

**B-03 並び順（E1/E2）**
- 合格: 設計通りのソート（happy_score 降順 → published_at 降順）

**B-04 サムネ失敗耐性（E1/E2）**
- 手順: ネットを一時的に落とす / 低速回線でスクロール
- 合格: プレースホルダーで表示が崩れない、リスト高さがガタつかない

**B-05 Pull to refresh（E1/E2）**
- 手順: 一覧の最上部で引っ張って更新（1回目・2回目）
- 合格: 失敗してもエラー表示＋再試行できる、成功したら一覧が更新される

---

### C. 記事詳細（SC-02）

**C-01 3行要約（E1/E2）**
- 手順: 2〜3本の記事を開いて要約を確認
- 合格: 3行整形ガードが効いており "3行" になっている、改行が効いて読みやすい
- NG: 要約が空 / 1行だけ / 巨大な1段落

**C-02 出典リンク（E1/E2）**
- 手順: 出典リンクをタップ
- 合格: 外部ブラウザで正しく開く
- NG: リンクが壊れている

**C-03 共有（E1/E2）**
- 手順: 共有ボタンをタップ → キャンセルして戻る
- 合格: OS共有シートが出る、キャンセルしても壊れない

---

### D. 保存（SC-03）

**D-01 保存/解除（E0可）**
- 手順: 一覧で保存トグル → 保存一覧（SC-03）を確認 → 解除して反映確認
- 合格: 保存/解除がUIに即時反映される
- NG: タップしても変化しない

**D-02 再起動後も保持（E0/E1/E2）**
- 手順: 保存 → アプリkill → 再起動 → 保存一覧を確認
- 合格: 保存が維持されている
- NG: 再起動で消える

---

### E. 設定・通知（SC-04/05）

**E-01 通知権限ON/OFF（E0/E2）**
- 手順: 通知ON → OS権限ダイアログを確認。拒否後の状態表示も確認。"通知設定を開く" でOS設定へ遷移確認
- 合格: 権限状態に応じて案内が正しい、OS設定へ誘導できる

**E-02 通知時刻HH:00（E2）**
- 手順: 次のHH:00に設定し、実際に届くか確認
- 前提: Scheduler稼働、バッテリー最適化OFF
- 合格: 設定時間（±数分許容）に通知が届く。OFF なら届かない

**E-03 通知タップ deeplink（E2）**
- 合格: Todayへ遷移する、空でもクラッシュしない、二重遷移しない

**E-04 連打/多重通知耐性（E2）**
- 合格: 連続タップしても落ちない、履歴が暴走しない

---

### F. ミュートワード（MVP）

**F-01 追加・フィルタ（E0/E1/E2）**
- 手順: ミュートワード追加（例: `disaster`）→ Today一覧が即座に非表示になるか確認 → 削除して復帰確認
- 合格: 追加で非表示、削除で復帰（クライアントフィルタ）

---

### G. 例外系

**G-01 DNS失敗（E0/E1/E2）**
- 合格: ユーザー向け表示は簡潔、ログに詳細、再試行導線あり

**G-02 タイムアウト/500（E1/E2）**
- 合格: 落ちない・再試行できる

## 2. バックエンド単体

### A. 日次バッチ

**BA-01 20本生成（E1/E2）**
- 手順:
  1. バッチを手動実行: `gcloud run jobs execute happynews-daily-batch --region asia-northeast1`
  2. Firestore `days/{day_key}` を確認: `article_ids` が **20件**
  3. `articles` が20件作成されていること
  4. `runs/{run_id}` の counts が妥当（fetched>0 / published=20）
- 合格: 20本が確定して入っている

**BA-02 候補上限ブレーキ（E1/E2）**
- 手順: `runs/{run_id}` の counts（fetched/deduped/filtered）を確認
- 合格: 500件を超えない（`candidate_hard_limit_per_day` で打ち切られる）

**BA-03 Rule Filter（E1/E2）**
- 手順: Firestore の `candidates` を確認
- 合格: NGワード・NGソースが弾かれ `rule_filtered=true` がセットされている

**BA-04 LLM分類（E1/E2）**
- 合格: `happy_score` / `category` / `tags` / `is_ng` が各 candidate に付与されている

**BA-05 3行要約整形ガード（E1/E2）**
- 合格: `articles` の要約が必ず3行になっている、禁止語が混入していない

---

### B. フォールバック

**BB-01 当日0件→latestへ（E1/E2）**
- 手順: 当日 `days/{today}` を削除（dev環境のみ）→ `GET /v1/days/latest` を叩く → `GET /v1/days/{today}/articles` を叩く
- 合格: 最大7日遡って latest が返る

**BB-02 7日分全て0件（E1/E2）**
- 合格: 空状態が穏当（クラッシュなし、エラーの説明あり）

---

### C. candidates 7日保持

**BC-01 TTLと追跡情報（E1/E2）**
- 手順: Firestore の `candidates` コレクションを確認
- 合格:
  - `ttl_delete_at` が入っている（datetime型）
  - `rule_filtered` / reasons / `llm_*` が残っている（「なぜ落ちた？なぜ採用？」が追跡できる）

---

### D. 監視・ログ

**BD-01 ログ追跡（E1/E2）**
- 合格: バッチ成功/失敗がログで追える
- 合格: `published < 20` のときにアラートまたはログで検知できる
- 合格: API 5xx / 遅延が Cloud Logging で確認できる

## 3. 統合テスト

### A. 通知 end-to-end（E2）

- 手順:
  1. アプリ起動（匿名ログイン完了）
  2. 通知 ON
  3. 次の HH:00 に設定
  4. その時刻に通知が届くか確認
  5. 通知をタップ → Today へ遷移するか確認
- 合格: 届く、タップで Today に飛ぶ（deeplink）

### B. 失敗系（壊れ方チェック）

**INT-B01 LLM/キーが無効（E2）**
- 手順: OPENAI_API_KEY をわざと外してバッチを実行（dev環境のみ）
- 合格: バッチは失敗するが、アプリは直近日を表示できる（壊れない）

**INT-B02 RSSソース障害（E1/E2）**
- 合格: 候補が減っても20本を作れる。作れない場合はログが明確

**INT-B03 ネットワーク不安定（E0/E1/E2）**
- 合格: アプリが落ちない、キャッシュが効く、再試行できる

## 4. テスト証跡フォーマット

バグ報告・修正依頼に含める情報:

- ケースID（例: B-02）
- 環境（E0/E1/E2）
- 期待した結果（1行）
- 実際の結果（1行）
- スクリーンショット or 画面録画
- `adb logcat` の関連箇所（30〜80行）
