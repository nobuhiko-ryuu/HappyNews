了解。実機テストは「Android（見える挙動）」と「バックエンド（日次で勝手に回る挙動）」を**分けて**、最後に「統合（通知・deeplink・フォールバック）」で締めるのが最短です。
下は **リリースまでに“必ず確認すべきチェックリスト”** をまとめたもの。

---

# 実機テスト総合チェックリスト（リリース前）

## 0) 事前準備（最初に一回だけ）

* [ ] Android：`dev/prod` の接続先が想定どおり（API Base URL）
* [ ] Android：Firebase（匿名Auth / FCM）が有効で、端末でネット接続OK
* [ ] Backend：`configs/global` が初期値どおり

  * candidate_target=200 / hard_limit=500 / publish_count=20 / fallback_max_days=7
* [ ] Backend：`sources` が 30〜60 件程度入り、`enabled=true` になっている
* [ ] Backend：Secrets（OPENAI_API_KEY / Firebase Admin / ほか）がprod相当で有効

---

## 1) Androidアプリ単体（ユーザー体験の核）

### A. 初回起動・ログイン

* [ ] 初回起動で匿名ログイン→Todayが表示される
* [ ] ログイン失敗時に “Failed→再試行” で復帰できる
* [ ] 端末の機内モード/圏外で起動しても「落ちない」「適切な案内」になる

### B. Today一覧（SC-01）

* [ ] 20本表示される（当日 or 直近日フォールバック）
* [ ] スクロールが軽い（カクつき・固まりがない）
* [ ] サムネが取得できない記事があってもプレースホルダーで破綻しない
* [ ] Pull to refresh が動作する（極端に遅くならない）

### C. 記事詳細（SC-02）

* [ ] タイトル・3行要約・タグ・出典が表示される
* [ ] 出典リンクタップで外部ブラウザが開く
* [ ] 共有がOS共有シートで出る

### D. 保存（SC-03）

* [ ] 一覧/詳細から保存トグルができる
* [ ] 保存一覧に反映され、再起動後も保持される
* [ ] 解除もできる

### E. 設定（SC-04/05/06）

* [ ] 通知 ON/OFF が効く
* [ ] 通知時刻（HH:00）変更が効く（保存される）
* [ ] OS通知権限が未許可のとき、案内と設定誘導が正しい
* [ ] 法務/情報画面が開き、リンクが正しい

### F. ミュートワード（MVP）

* [ ] ミュートワードの追加/削除ができる
* [ ] ミュート対象が一覧で非表示になる（クライアント側フィルタ）

---

## 2) バックエンド単体（日次運用の核）

### A. 日次バッチ（Daily Batch）

* [ ] `sources(enabled)` から収集して候補が作られる
* [ ] 候補収集が 200 を目標に進み、500 を超えたら打ち切る（ブレーキ確認）
* [ ] Rule Filter が働く（NGワード/NGソースが弾かれる）
* [ ] LLM分類が動く（happy_score/category/tags/is_ng が付く）
* [ ] ランキングで 20本が選定される（カテゴリ偏りが極端にならない）
* [ ] 要約が**必ず3行**になっている（禁止語も混入しない）
* [ ] publish が成功し `days/{day_key}` に **20本**入る

### B. フォールバック

* [ ] 当日分が無い/バッチ失敗時に `latest` が直近日（最大7日）を返す
* [ ] 当日0件でもアプリが「直近日を表示」できる

### C. candidates 7日保持（運用改善の核）

* [ ] candidates に `ttl_delete_at` が入っている（datetime）
* [ ] candidates に rule/classify の結果が残る（原因追跡できる）
* [ ] Firestore TTL設定が有効（後日、自然に消える）

### D. 監視・ログ

* [ ] バッチ成功/失敗がログで追える
* [ ] 掲載数<20、NG混入>0 があればアラート（または最低限ログで検知）
* [ ] APIの5xxや遅延が見える

---

## 3) 統合テスト（リリース前に必須）

### A. 通知 end-to-end

* [ ] FCMトークンがバックエンドに登録される
* [ ] 通知ONにして、設定したHH:00に通知が届く
* [ ] 通知タップで Today に確実に遷移する（deeplink）
* [ ] 通知OFFで届かないことを確認

### B. 代表的な失敗系（壊れ方チェック）

* [ ] LLM/APIキーが無効のとき：バッチが落ちてもアプリは直近日を出す（壊れない）
* [ ] RSS元が落ちている：候補が減っても20本を作れる/作れない時のログが明確
* [ ] ネットワークが不安定：アプリが落ちない・キャッシュが効く

---

## 4) リリース直前チェック（短いけど重要）

* [ ] 本番のSecrets/設定値がprodに入っている（devを向いてない）
* [ ] `configs/global` のNGワード/カテゴリ上限が過激すぎない（20本埋まらなくなる）
* [ ] 1日分の運用を“手動で回せる”手順がある（day_key指定・再実行）
* [ ] ストア文言/法務リンク/問い合わせ先が正しい
