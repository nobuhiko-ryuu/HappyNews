## 1) 実機テストの前提条件（環境の型を決める）

### 環境は4段階で考える（おすすめ）

**E0：アプリ単体（バックエンド無し）**

* 目的：起動・画面遷移・UI崩れない・エラーが穏当
* 期待：Todayは取得失敗になるのは“自然”。ただし**クラッシュしない**のが合格。

**E1：ローカルバックエンド（Docker + Firestore Emulator）**

* 目的：API疎通、days取得、フォールバック、候補TTLなどを安く速く検証
* 注意：**実機からPCへの接続**が必要（localhost/10.0.2.2はダメ）

**E2：Staging（Cloud Run + Firestore + Scheduler/FCM）縮小**

* 目的：E2E（バッチ→days生成→通知→deeplink）を本番に近い形で確認
* 条件：候補50/日・上限100・ソース5〜10など縮小（あなたの引き継ぎメモ通り）

**E3：Prod（リリース前最終）**

* 目的：最終ゲート（Secrets/監視/コスト/ストア文言/法務リンク）

---

## 2) 端末・ビルドの共通前提（全部の環境で必要）

### 端末

* Android **minSdk以上**（たぶん26=Android 8.0+）
* Wi-Fi推奨（通信量/安定性のため）
* 省電力の例外設定：HappyNewsを **バッテリー最適化しない**（通知試験の必須）

### Firebase（Android側が起動する最低条件）

* `android/app/google-services.json` 配置済み
* appに `com.google.gms.google-services` plugin 適用済み
* Firebase Consoleで **Anonymous Auth 有効化**（`signInAnonymously()`するため）

### ネットワーク（E1で重要）

* 実機→PCにHTTPで当てる場合、Android側が **cleartext許可**してないと失敗します

  * 失敗例：`CLEARTEXT communication not permitted`
  * 対策：debugだけ `usesCleartextTraffic=true` か network security config

---

## 3) 環境ごとの構築要件（最低限）

### E0：アプリ単体（バックエンド無し）

* 何も立てない
* 合格条件：Todayで通信エラーが出ても **クラッシュしない／再試行できる／UIが崩れない**

### E1：ローカル（Docker）

* PCで `backend/docker-compose.yml` を起動（API + Firestore Emulator）
* seed → batch → API確認（あなたの手順でOK）
* Androidの `BASE_URL` を **PCのLAN内IP** に向ける

  * 例：`http://192.168.0.10:8080`
  * `localhost` / `10.0.2.2` は実機では不可

### E2：Staging

* Cloud Run / Firestore / Scheduler / FCM が設定済み
* Schedulerは「正時待ち」がダルいので、可能なら

  * Scheduler “Run now” 相当（手動実行）
  * もしくは stagingだけテスト用トリガー（最小権限）
    で時短する

## 4) テストパターン（前提条件つき）

下は「どの環境でやるか」も明記してある版。
（IDで管理すると、バグ報告が一気に速くなります。）

### A. 起動・認証（E0/E1/E2）

**A-01 起動スモーク**（E0）

* 前提：Firebase設定済み
* 手順：起動 → ホーム表示まで
* 期待：クラッシュしない

**A-02 匿名ログイン成功**（E0/E1/E2）

* 手順：起動直後、認証待機→成功
* 期待：Today取得に進む（失敗しても再試行導線がある）

**A-03 認証失敗→再試行で復帰**（E0/E1/E2）

* 手順：機内モードONで起動→失敗→OFF→再試行
* 期待：復帰する（アプリ再起動なしで）

---

### B. Today一覧（E0/E1/E2）

**B-01 Today表示（成功）**（E1/E2）

* 前提：`days/{day_key}` に20本入っている（seed or batch済み）
* 期待：20本表示、スクロール快適、サムネ遅延ロードOK

**B-02 Today表示（失敗時UI）**（E0）

* 前提：バックエンド無し or BASE_URL不達
* 期待：`Unable to resolve host` が出ても

  * 画面が壊れない
  * “取得できません/再試行” 表示
  * Pull-to-refreshで再試行できる
    （ここがダメならUX修正対象）

**B-03 並び順**（E1/E2）

* 期待：想定のソート（例：スコア降順→新しさ等、設計通り）

**B-04 サムネ失敗耐性**（E1/E2）

* 手順：ネットを一瞬落とす/遅い回線でスクロール
* 期待：プレースホルダーで崩れない、リスト高さガタガタしない

---

### C. 詳細画面（E1/E2推奨、E0でも一部可）

**C-01 3行要約固定**（E1/E2）

* 期待：3行整形ガードが効いて“3行”になっている

**C-02 出典リンク**（E1/E2）

* 期待：外部ブラウザで正しく開く

**C-03 共有**（E1/E2）

* 期待：共有インテントが開き、URL等が正しい

---

### D. 保存（E0/E1/E2）

**D-01 保存/解除**（E0でも可）

* 期待：保存状態がUIに反映

**D-02 再起動後も保持**（E0/E1/E2）

* 手順：保存→アプリkill→再起動
* 期待：保持される（ローカル永続 or Firestore、仕様通り）

---

### E. フォールバック（E1/E2）

**E-01 当日0件→latestへ**（E1/E2）

* 前提：当日daysが0、過去に在庫あり
* 期待：最大7日まで遡って表示

**E-02 7日分0件**（E1/E2）

* 期待：空状態が穏当（クラッシュなし、説明あり）

---

### F. 通知（E2で本命、E0/E1は部分）

**F-01 通知権限ON/OFF**（E0/E2）

* 期待：OS設定へ誘導できる、状態表示が破綻しない

**F-02 正時に届く（HH:00）**（E2）

* 前提：Scheduler稼働、バッテリー最適化OFF
* 期待：正時付近で受信

**F-03 通知タップdeeplink**（E2）

* 期待：Todayへ遷移、空でもクラッシュしない、二重遷移しない

**F-04 連打/多重通知耐性**（E2）

* 期待：連続タップでも落ちない、履歴が暴走しない

---

### G. 例外系（E0/E1/E2）

**G-01 DNS失敗（Unable to resolve host）**（E0/E1/E2）

* 期待：ユーザー向け表示は簡潔、ログに詳細、再試行導線あり
  （“OKか？”の答え：**挙動がこの形ならOK、赤裸々エラー垂れ流しなら修正対象**）

**G-02 タイムアウト/500**（E1/E2）

* 期待：同上（落ちない・再試行できる）

---

## 5) 実機テストの「証跡」フォーマット（バグ修正が速くなる）

各ケースでこれを残すと、Claude Codeへの修正指示が一瞬で作れます。

* ケースID（例：B-02）
* 前提環境（E0/E1/E2）
* 期待/実際（1行ずつ）
* スクショ or 画面録画
* `adb logcat` 30〜80行（該当箇所）

## 6) いまのあなたの状況への当てはめ

あなたは今「起動OK → Todayで `Unable to resolve host`」なので、たぶん環境は **E0**（バックエンド未構築）です。
この段階での合格ラインは：

* エラーは出てもいい
* ただし **ホームが壊れない / 再試行できる / UXが穏当** が必須

もし、ホームが“生ログそのまま表示”みたいになってるなら、そこは修正対象にします（UIを整える）。
