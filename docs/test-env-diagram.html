🗺️ HappyNews テスト環境の関係図

Google Cloud / Firebase / Docker の役割と、テストの「レベル」を整理した地図

📦 Part 1 ── 本番アプリはどう動いているか？

🐳 Docker（コンテナ）

「お弁当箱」のようなもの。アプリと必要なライブラリをひとまとめに詰め込んで、
どのパソコンやサーバーでも同じように動かせる技術。

📄Dockerfile
「お弁当の作り方レシピ」。
Python のインストール・コードのコピーなど、手順が書いてある。
🖼️Docker イメージ / コンテナ
レシピを使って作った「お弁当」= イメージ。
それを実際に開けて食べている状態 = コンテナ（実行中）。
⬇️Docker コンテナが動く場所
☁️ Google Cloud Platform（GCP）

「Googleが持っている超巨大なコンピュータ貸し出しサービス」。
サーバーを自分で買わなくても、必要なぶんだけ借りて使える。

🏃Cloud Run
Docker コンテナを
クラウドで走らせる場所。
バックエンド API がここで動く。
Docker必須
🗄️Firestore
Google の NoSQL データベース。
記事・ユーザー・設定などを保存する「引き出し」。
DB
⏰Cloud Scheduler
「毎朝6時に自動実行」などを設定できる。日次バッチのタイマー役。
定期実行
🔑Secret Manager
APIキーなど秘密の情報を安全に保管する金庫。
セキュリティ
⬇️Firebase は GCP の上に乗っている
🔥 Firebase（GCP の上に乗っているサービス群）

Firebase は GCP の子供。「モバイルアプリ向けに使いやすくした GCP のセット」。
Firestore は Firebase でも GCP でも使える（同じもの！）。

👤Firebase Authentication
ユーザーログインの仕組み。
HappyNews では「匿名ログイン」を使っている（インストールだけで使える）。
Android連携
📬FCM（Firebase Cloud Messaging）
Android への「プッシュ通知」を送る仕組み。
「今日のハッピーニュース届きました」の通知はここ経由。
プッシュ通知
🗄️Firestore（再掲）
Firebase 経由でも GCP 経由でも同じデータベースにアクセスできる。Android も Backend も同じ DB を使う。
GCPと同じ

🧪 Part 2 ── テストには「3つのレベル」がある

テストは「本物を使うほど信頼度は上がるが、コスト・時間・準備が増える」というトレードオフがある。
HappyNews では 普段は外部ゼロ（速い・無料）、夜だけ本物（信頼度チェック）という戦略。
🧪 テストの3レベル
🟢レベル1 ── ユニットテスト / スタブテスト 外部ゼロCI毎PR実行
テストコード
→
スタブ（偽物の外部）
→
結果確認
スタブ＝本物のAPIや DB の代わりに「常に決まった答えを返す偽物」。
インターネット不要・お金かからない・一瞬で終わる。
例：test_batch.py, test_notify.py（EXTERNAL_MODE=stub）
🟡レベル2 ── エミュレーターテスト Emulator必要ローカルのみ
テストコード
→
Firestore Emulator
（localhost:8080）
→
結果確認
Firestore Emulator＝「本物の Firestore を自分のパソコンの中で再現したもの」。
本物と同じ動きをするが、お金はかからず、データも本番に影響しない。
Docker で起動するか、Firebase CLI（firebase emulators:start）で起動する。
例：test_api_emulator.py（nightly/ フォルダ）→ CI では --ignore で除外中
🔴レベル3 ── 本物GCPテスト（ナイトリー） 実費あり毎夜JST05:00
テストコード
→
本物 GCP
Firestore / OpenAI
→
結果確認
本物のクラウドと本物のAIを使って動かす「最終チェック」。
OpenAI APIキー・GCPの認証情報が必要（GitHub Secrets に登録）。
失敗しても CI を落とさない（|| true）のは、ネットワーク障害など外部要因のブレがあるため。
例：nightly.yml が毎夜自動実行（EXTERNAL_MODE=real）

🚀 Part 3 ── Firestore Emulator をどうやって起動するか？

方法A：Firebase CLI
Node.js + Firebase CLI を使う
# インストール（一度だけ）
npm install -g firebase-tools

# emulator 起動
firebase emulators:start \
  --only firestore \
  --project demo-happynews
これだけで localhost:8080 に Firestore が立ち上がる。
方法B：Docker Compose
Docker だけで完結させる
# docker-compose.yml に書けば
# docker compose up だけで起動できる

services:
  firestore-emulator:
    image: gcr.io/google.com/cloudsdktool/...
    ports:
      - "8080:8080"
Node.js 不要・CI との親和性が高い。
今後 CI に組み込むならこちらが便利。

🎯 3行まとめ
① Docker ＝アプリを詰め込む「お弁当箱」。Cloud Run（GCP）の上で走る。
② Firebase ＝GCPの子供セット。Auth（ログイン）・FCM（通知）・Firestore（DB）をスマホアプリ向けに使いやすくしたもの。
③ テストのレベル ＝ スタブ（偽物・速い）→ Emulator（本物に近い・ローカル）→ 本物GCP（最終確認・夜のみ）の3段階で信頼度を積み上げる。
